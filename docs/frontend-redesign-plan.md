# 前端重设计计划 — AI 语音绘图助手

## 1. 背景与动机

### 1.1 当前状态

项目已完成核心链路（语音→Agent→Canvas），但交互体验存在明显问题：

| 问题 | 现状 | 影响 |
|------|------|------|
| 语音触发笨重 | 必须说"开始绘画"/"结束绘画"关键词 | 像在对机器人喊口令，不自然 |
| 零视觉反馈 | 顶部一行小状态文字 | 用户不知道系统在做什么 |
| 无兜底手段 | 识别错了只能语音再说一遍 | 无法撤销、无预览、无确认 |
| UI 简陋 | 单一暗色主题，无动画 | 不像一个产品 |
| 旧组件散落 | VoiceInput/TextInput/DebugPanel 不用但留在代码中 | 混乱 |
| 关键词启动 | 必须说咒语才能进入绘图模式 | 认知负担高 |

### 1.2 参考项目启发

从 `voiceDraw` 项目的设计中借鉴以下模式：

- **持续对话循环**：点一次麦克风 → 持续对话 → TTS 反馈 → 自动继续监听
- **流式状态显示**：用户始终知道系统在"听/理解/执行/完成"
- **JSON 解析鲁棒性**：多层回退 + 自动重试
- **意图分类前置**：本地判断是否为绘图指令，避免无效 LLM 调用

## 2. 交互模式设计

### 2.1 语音主导模式（第一期实现）

核心理念：**手动开始 + 语音结束，始终监听无需关键词**

```
                    ┌──────────────────────┐
                    │  🎤 点击开始绘画     │  ← 手动按钮启动
                    └──────────┬───────────┘
                               │ 进入绘图模式
                               ▼
              ┌────────────────────────────────┐
              │  🟢 正在听...                  │  ← 始终监听，无关键词
              │  "画一个红色的圆"              │
              └──────────┬─────────────────────┘
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
    绘图指令         控制指令        模糊/闲聊
    → LLM 处理      → 本地执行       → TTS 提示
    → Canvas 渲染   → 撤销/清空      → 不调 LLM
    → TTS 反馈      → TTS 反馈
          │              │              │
          └──────────────┴──────────────┘
                         │
                         ▼
              ┌────────────────────────────────┐
              │  继续监听下一句...              │
              │  说"结束绘画"退出              │  ← 语音结束，不点按钮
              └────────────────────────────────┘
```

### 2.2 与纯语音模式的区别

| | 当前（纯语音） | 目标（语音主导） |
|------|------|------|
| 进入绘图 | 说"开始绘画" | 点击麦克风按钮 |
| 退出绘图 | 说"结束绘画" | 说"结束绘画"（语音） |
| 发送指令 | 说 + 1.5s 静默 | 说 + 1.5s 静默 |
| 兜底操作 | 无 | 快捷栏（撤销/清空） |

### 2.3 未来扩展（第二期）：双模式切换

在顶部加一个 `[纯语音] [语音主导]` 的切换开关：

- **纯语音模式**：完全无按钮，通过"开始绘画"/"结束绘画"关键词控制（当前 VoiceController 的升级版）
- **语音主导模式**：手动启动 + 快捷栏辅助（本期实现）

## 3. TTS 语音反馈说明

### 3.1 什么是 TTS

TTS（Text-to-Speech）即文字转语音。浏览器内置了 `window.speechSynthesis` API，可以免费将文字用中文朗读出来。例如：

```javascript
const utter = new SpeechSynthesisUtterance('画了一个红色的圆')
utter.lang = 'zh-CN'
speechSynthesis.speak(utter)
```

### 3.2 TTS 在绘图流程中的作用

```
用户说"画红色圆"
    ↓
ASR 识别 → LLM 处理 → Canvas 渲染 ✓
    ↓
TTS 播报："好的，画了一个红色圆" ← 用户不用看屏幕也知道完成了
    ↓
自动继续监听下一句
```

作用：**听觉确认机制**。用户无需盯着屏幕，语音播报告诉你结果。

### 3.3 TTS 反馈策略

| 场景 | TTS 内容 | 时机 |
|------|---------|------|
| 绘图成功 | "好的" 或 "画了一个红色的圆" | Canvas 渲染后 |
| 绘图优化 | "已优化：画了一个小圆" | 模糊指令被自动调整后 |
| 无法理解 | "没理解，请再说一遍" | LLM 返回 error |
| 撤销 | "已撤销" | 本地执行 |
| 清空 | "画布已清空" | 本地执行 |
| 结束 | "绘画已结束" | 退出绘图模式 |

### 3.4 注意事项

- 语音播报期间，新的用户语音会**打断**当前播报（`speechSynthesis.cancel()`）
- 非 Chrome/Edge 浏览器可能音色有限
- 可以后续扩展音色选择（参考 voiceDraw 的音色切换）

### 5.2 本期新增

| 类别 | 示例指令 | 处理方式 | 优先级 |
|------|---------|---------|--------|
| 本地撤销 | "撤销" / 点击撤销按钮 | 本地 layers.pop() + Canvas 重渲染 | P0 |
| 本地清空 | "清空" / 点击清空按钮 | 本地 layers = [] + Canvas 清空 | P0 |
| 本地意图分类 | 所有输入先分类 | intentClassifier 判断→分发给 LLM 或本地 | P0 |
| 语音结束 | "结束绘画" | VoiceController 退出绘图模式 | P0 |
| 开始确认 | 点击麦克风按钮 | 进入持续监听模式 | P0 |
| TTS 确认 | 绘图成功后语音播报 | tts.js speak(summary) | P1 |

### 5.3 未来规划（未完成）

| 功能 | 预期能力 | 未完成原因 |
|------|---------|-----------|
| 撤销多步栈 | 支持多次撤销 | Canvas 状态栈实现复杂度高，当前图层系统不支持 incremental undo |
| 纯语音模式 | 完全无按钮交互 | 本期先聚焦语音主导模式，下期实现双模式切换 |
| 画布导出 | "保存"/"导出" → 下载 PNG | P2 优先级，需要 Canvas.toDataURL 封装 |
| 指令预览撤销 | 2s 窗口期内说"不对"取消 | P2 优先级，核心交互稳定后再做 |
| 云端 TTS | 接入 Azure Speech | 浏览器 TTS 音色有限，但增加外部依赖和成本 |
| 画布持久化 | 刷新保留 | localStorage 存储 layers 数组，恢复时需考虑 Fabric.js 对象序列化 |

## 6. 容错设计

### 6.1 语音识别层

| 场景 | 处理 |
|------|------|
| 1.5s 静默 | 自动提交当前文本，重新开始监听 |
| 15s 无语音 | 自动重启语音识别（Web Speech API 偶发停止） |
| 识别错误（not-allowed） | 提示用户在浏览器设置中允许麦克风 |
| 其他识别错误 | 提示 + 自动重试 |
| 权限恢复检测 | 监听 `permissions.query` 变化，权限恢复后自动启动 |

### 6.2 意图分类层

```
用户输入文本
    ↓
intentClassifier(text) →
    ├── 'draw'    → 发 LLM 处理
    ├── 'control' → 本地执行（撤销/清空）
    └── 'chat'    → TTS 提示"请说绘图指令"（不调 LLM）
```

分类规则：
- 包含绘画动词（画/绘制/加/添）→ `draw`
- 包含控制词（撤销/回退/清空/清除/删）→ `control`
- 其他 → `chat`（闲聊，不消耗 API）

### 6.3 JSON 解析层

LLM 返回 JSON 的鲁棒提取策略（参考 voiceDraw）：

1. 优先提取 `<output>` 标签内容
2. 正则匹配 `{...}` JSON 对象
3. 字符串感知括号匹配（正确处理 JSON 字符串内的 `{}` 和转义引号）
4. 失败自动重试（最多 3 次，每次给更具体的格式提示）
5. 全部失败 → 返回 `status: "error"`

### 6.4 网络/服务异常

| 异常 | 前端行为 |
|------|---------|
| 代理不可用 | catch → TTS "出错了，请重试" |
| LLM 返回非 JSON | 重试 3 次 → error status |
| API Key 无效 | 401 → 提示配置错误 |
| 网络超时 | AbortController 15s 超时 → 提示重试 |

## 7. UI 设计方向（第三阶段）

### 7.1 设计目标

- **现代感**：玻璃态 + 柔和阴影 + 渐变
- **状态可视化**：用颜色和动画明确表达"听/想/画/完"四态
- **空间利用**：画布为主角，工具栏不抢视线

### 7.2 配色方案

```
背景：深蓝灰 #0f0f23
卡片：半透明 #1a1a3e / rgba(255,255,255,0.05)
主色调：紫罗兰 #7c5cfc → 浅紫 #9b7eff
成功态：翠绿 #4ade80
思考态：琥珀 #fbbf24
错误态：玫红 #f87171
画布：纯白 #ffffff
文字：浅灰 #e0e0e0 / 中灰 #888
```
