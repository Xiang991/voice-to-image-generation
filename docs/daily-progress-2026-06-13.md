# AI 语音绘图助手 — 2026年6月13日 进度总结

## 今日改进总览（产品经理视角）

### 一、核心体验：从"关键词咒语"到"自然语音交互"

**改进前**：用户必须说"开始绘画"进入模式 → 说绘图指令 → 说"结束绘画"退出。本质是"对机器人喊口令"，认知负担高，不自然。

**改进后（设计阶段）**：用户点击一次麦克风按钮 → 持续对话。说绘图指令直接执行 → TTS 语音确认 → 自动继续监听下一句。说"结束绘画"退出（语音结束，不用手）。全程自然对话。

**关键设计决策**：

- 去掉"开始绘画"关键词 → 改为手动点击按钮启动（避免误触发 + 用户主动控制启动时机）
- 保留"结束绘画"语音 → 结束时用户可能双手已离开键盘，语音结束比找按钮更快
- 始终监听 + 1.5s 静默自动提交 → 无需关键词分割每条指令

### 二、交互模式：语音主导（非纯语音）

**设计决策**：之前"零按钮纯语音"的方案是错误的。原因：

1. 纯语音模式没有兜底——识别错只能再说一遍，无法快速撤销
2. 用户首次进入时不知道该说什么——没有视觉引导
3. 按钮是"安全网"——降低用户焦虑，知道"即使语音失败，还能点按钮"

**语音主导模式** = 语音作为主要输入 + 按钮作为辅助兜底：

- 主要操作：语音指令绘图
- 辅助操作：快捷栏按钮（撤销/清空）
- 启动：手动点击麦克风按钮
- 结束：说"结束绘画"（语音）

**未来扩展**：顶部加 `[纯语音] [语音主导]` 模式切换开关。纯语音模式适用于演示/沉浸式创作；语音主导模式适用于日常使用。

### 三、指令路由：本地意图分类前置

**改进前**：所有语音文本直接发给 LLM，包括"你好"、"撤销"、"清空"。造成不必要的 API 调用成本和延迟。

**改进后**：前端本地意图分类：

```
用户文本 → intentClassifier →
  ├── 'draw'    → 发 LLM 处理（~1-3s）
  ├── 'control' → 本地执行（<50ms + TTS）
  └── 'chat'    → TTS 提示"请说绘图指令"（不调 LLM）
```

**收益**：
- 撤销/清空延迟从 1-3s 降到 <50ms
- 闲聊不消耗 API 费用
- 本地执行的控制指令更可靠（不依赖 LLM 理解）

### 四、语音反馈：TTS 听觉确认闭环

**什么是 TTS**：浏览器内置的文字转语音（`window.speechSynthesis`），免费，无需额外服务。

**为什么需要**：用户盯着画布绘图时，语音确认让用户不用移开视线。绘图是视觉任务，确认应该是听觉反馈——互补通道。

**反馈策略**：

| 场景 | TTS 反馈 | 目的 |
|------|---------|------|
| 绘图成功 | "好的，画了一个红色圆形" | 确认执行 |
| 指令优化 | "已优化：画了一个小圆" | 告知调整 |
| 无法理解 | "没理解，请再说一遍" | 引导重试 |
| 撤销 | "已撤销" | 确认操作 |
| 清空 | "画布已清空" | 确认操作 |
| 闲聊 | "请说绘图指令" | 引导聚焦 |

### 五、组件架构清理

**改进前**：6 个组件文件，其中 3 个（VoiceInput、TextInput、DebugPanel）不渲染但保留在 src/components/ 中，造成混淆。

- 三个旧组件移入 `src/components/_archived/` 暂存

## 已知待解决问题

- **server/ 代理层未实现**：当前 API Key 仍暴露在前端 `.env` 中。Step 7（薄代理）在 progress.json 中标记为 pending，但本次前端重设计暂不涉及，需后续处理。
- **画布摘要模块未实现**：`canvasSummary.js` 不存在，LLM 目前不感知画布上已有什么图形。复合指令的空间布局完全依赖 LLM 自身判断。
- **撤销仅支持单步**：当前 `layers.pop()` 只回退最后一步，不支持多步撤销栈。
- **SVG 质量依赖 LLM 能力**：DeepSeek V4 Flash 生成的 SVG 质量有限，复杂物体可能效果差。
- **TTS 音色依赖浏览器**：不同 OS/浏览器的中文音色质量和数量差异大。Chrome 中文音色好于 Firefox/Safari。
- **纯语音模式暂未实现**：本期聚焦语音主导模式，纯语音（零按钮）模式后续以双模式切换方式加入。
